Bigfloat: Arbitrary-precision floating-point numbers and functions

Neil Toronto <neil.toronto@gmail.com>

 (require "../bigfloat-typed.rkt")

1 Introduction

Bigfloat is a pure Racket library for representing and manipulating floating-point numbers with any precision.

Eventually, all of its operations will be verified, through deterministic and randomized testing, to compute the right answers and correctly round them when they can’t be exactly represented. Curently, most operations have passed thousands of randomized tests up to 16k bits precision. Arithmetic, square root, comparisons and conversions additionally pass exhaustive tests up to 10 bits precision, and use algorithms that are proven correct. The constants bf-pi, bf-e, bf-log-2 and bf-log-10 pass exhaustive tests up to 65k bits precision.

Bigfloat is written in Typed Racket but can be used naturally in untyped programs.

2 Quick start

  1. Require the Bigfloat module using (require "bigfloat-typed.rkt") or (require "bigfloat-untyped.rkt").

  2. Set the working precision using (bf-bits <some-number-of-bits>).

  3. Convert any Real value or properly formatted String to a bigfloat using (bf <some-real-or-string>).

  4. Operate on bigfloat values using “bf”-prefixed functions like bf+ and bfsin.

  5. Convert bigfloats to Real values using bigfloat->rational, bigfloat->float, and bigfloat->integer. Convert them to Strings using bigfloat->string.

To see some weakly motivated examples, continue on to the FAQ.

3 Fictionally Asked Questions

3.1 Why use Bigfloat?

There are a few reasons.

Reason: To calculate, with arbitrary precision, irrational numbers and irrational functions of floating-point numbers.

Examples:

  > (bf-bits 16)
  > (bfsqrt (bf 3))

  - : bigfloat

  (bf #e1.73206)

  > (bf-bits 179)
  > (bf-pi)

  - : bigfloat

  (bf #e3.141592653589793238462643383279502884197169399375105819)

A flonum has a 53-bit significand (we’ll say it has 53 bits of precision) and an 11-bit exponent. A bigfloat has an arbitrary precision of at least 2 bits and an unbounded exponent range.

Reason: To compute ridiculously large or small numbers with some degree of confidence.

Examples:

  > (bf-bits 64)
  > (bfexp (bfexp (bfexp (bf 4))))

  - : bigfloat

  (bf "3.20287203238572512202e+223593715528665832488403")

  > (bflog (bflog (bflog (bfexp (bfexp (bfexp (bf 4)))))))

  - : bigfloat

  (bf #e4.0)

* Be careful with repeated exponentiation! The result of (bfexp (bfexp (bfexp (bf 100)))) has an exponent that won’t fit in memory on anyone’s computer—it would take millions of yottabytes. Bigfloat will try things like this anyway, and eventually run out of memory. The exponent in the result of (bfexp (bfexp (bfexp (bf 4)))) is well outside the exponent range of almost all floating-point libraries. Additionally, the result can’t fit in memory as an exact rational.*

Reason: To verify your floating-point hardware.

IEEE 754-2008 stipulates that conforming implementations must correctly round the results of all operations. Roughly speaking, results can’t be more than half a bit off, where the bit in question is the least significant in the significand.

Of course, implementations that don’t adhere to the standard may not be so accurate, and implementations that claim to adhere to the standard may not, either.

You can use Bigfloat to verify the floating-point implementation on your CPU. For example, on my laptop, evaluating (exp 400) results in 5.221469689764346e+173. Note the last four decimal digits in the significand: 4346. Bigfloat says the answer should be

Examples:

  > (bf-bits 53)
  > (bigfloat->float (bfexp (bf 400)))

  - : Float

  5.221469689764144e+173

So the last four decimal digits should be 4144. On the other hand, my desktop computes 5.221469689764144e+173 as it should.

3.2 When shouldn’t I use Bigfloat?

Speed: Bigfloat functions can be hundreds to thousands times slower than flonum functions.

That’s not to say that they’re inefficient. For example, bflog implements the algorithm with the best known asymptotic complexity. It just doesn’t run directly in silicon, and it can’t take shortcuts that implementations using fixed precision can take.

3.3 Why don’t you just make a foreign function interface with [my favorite C/C++ numeric package]?

I have an incomplete FFI to MPFR that I plan to finish. I will likely use it as a Bigfloat backend when it is installed, to compute results that have exponents within its range.

3.4 Why don’t you prove that the results are correctly rounded?

Some algorithms are proved to round correctly, such as those for conversion to and from Real, for arithmetic, and for square root. Some could be with a little more work, such as the algorithm for bf-pi.

For the rest: it takes a lot of time to discover and prove epsilon bounds, to wrap things in Ziv loops, and to determine all the exact cases so the loops terminate.

However, it’s something that can be done independently and piecemeal, so anyone is welcome to do it and submit patches. Fortunately, the theoretical work has already been done by the fine folks who make MPFR, and many others. It would be nice to see it implemented functionally.

3.5 Why is the last digit of (bf-pi) not rounded correctly?

All bits but the last should be correct, and the last bit should be correctly rounded. This doesn’t guarantee that the last digit will be.

A decimal digit represents, at most, log(10)/log(2) bits, which is about 3.3. Because the decimal/bit boundary never lines up except at the decimal point, the last decimal digit of any bigfloat must represent fewer than 3.3 bits. So the last digit of a bigfloat’s decimal string is more often wrong than not, and it’s not a problem.

3.6 Why are there junk digits on the end of (bf 1.1) when it’s printed?

That’s approximately the value of flonum 1.1. Prefixing 1.1 with #e will make it a rational, equivalent to (bf 11/10), making most of the junk go away. In general, use exact rationals to create decimal constants.

4 Bigfloat API

(bf-bits) → Exact-Positive-Integer
(bf-bits precision) → void?
  precision : Exact-Positive-Integer
A parameter that determines the precision of new bigfloat values. Return values from Bigfloat library functions are bigfloats with (bf-bits) precision, no matter what the precisions of the inputs.* * There are a handful of exceptions to this rule, and none of them are standard mathematical functions. The exceptions are noted in this manual.

This parameter has a guard that ensures (bf-bits) > 1 because one-bit floats can’t be correctly rounded.

(struct bigfloat (sig exp))
  sig : Integer
  exp : Integer
* If you must construct a floating-point number from scratch, consider using new-bigfloat, which rounds the result to (bf-bits) bits. Represents an arbitrary-precision floating-point value. Generally, you wouldn’t construct bigfloats directly, but use bf instead to convert Real and String values*.

There are no bigfloats corresponding to flonums +inf.0, -inf.0 and +nan.0. Because exponents are unbounded, infinity rarely comes up. Bigfloat library functions raise exceptions instead of returning NaNs.

A bigfloat consists of a significand and an exponent, both signed. If x = (bigfloat s e), its rational value is (* s (expt 2 e)). Its precision is (integer-length (abs s)), or (bigfloat-bits x).

4.1 Construction and conversion

(bf value) → bigfloat
  value : (U String Real)
Constructs a bigfloat from a String or a Real.

Examples:

  > (bf-bits 128)
  > (bf 4)

  - : bigfloat

  (bf #e4.0)

  > (bf 1/7)

  - : bigfloat

  (bf #e0.1428571428571428571428571428571428571426)

  > (bf 41/10)

  - : bigfloat

  (bf #e4.1)

  > (bf "15e200000000000000")

  - : bigfloat

  (bf "1.5000000000000000000000000000000000000025e+200000000000001")

In the last example, the result of (bf "15e200000000000000") is displayed as a string conversion because the exact rational number is too large. (String conversions are displayed when the exponent magnitude is greater than 10000 decimal.)

Converting from flonum constants is generally a bad idea because flonums have only 53 bits precision. Prefix flonum constants you pass to bf with #e to turn them into Exact-Rational constants.

(bf2^ i) → bigfloat
  i : Integer
Equivalent to (bf (expt 2 i)) and (bfexp2 (bf i)), but much faster.

Examples:

  > (bf2^ 4)

  - : bigfloat

  (bf #e16.0)

  > (bf2^ -4)

  - : bigfloat

  (bf #e0.0625)

(bigfloat->integer x) → Integer
  x : bigfloat
(bigfloat->float x) → Float
  x : bigfloat
(bigfloat->rational x) → Exact-Rational
  x : bigfloat
Convert bigfloats to Integer, Float, and Exact-Rational values.

Only bigfloat->rational returns an exact value; bigfloat->integer truncates x first, and bigfloat->float rounds x to 53 bits precision first.

Examples:

  > (bf-bits 64)
  > (bigfloat->integer (bf 21/10))

  - : Integer

  2

  > (define x (bf 1/7))
  > (bigfloat->float x)

  - : Float

  0.14285714285714285

  > (bigfloat->rational x)

  - : Exact-Rational

  10540996613548315209/73786976294838206464

  > (rationalize (bigfloat->rational x) (expt 2 (- (bf-bits))))

  - : Exact-Rational

  1/7

  > (bf= x (rational->bigfloat (bigfloat->rational x)))

  - : Boolean

  #t

Be careful with bigfloat->integer and bigfloat->rational. It can be easy, especially when using exponentials, to create a bigfloat value that won’t fit in memory as an Integer or Exact-Rational.

(bigfloat->string x) → String
  x : bigfloat
Converts bigfloats to Strings.

The string includes enough digits that string->bigfloat reconstructs the bigfloat. It will be in scientific notation when the number is large or small enough.

Examples:

  > (bigfloat->string (bf 4))

  - : String

  "4.0"

  > (bigfloat->string (bf 1/10000))

  - : String

  "0.0001000000000000000000000000000000000000001"

  > (bigfloat->string (bf 1/100000000000000000000000000))

  - : String

  "9.99999999999999999999999999999999999999e-27"

  > (bigfloat->string
     (bf 10000000000000000000000000000000000000000000))

  - : String

  "1.0e+43"

(bigfloat-fields->string x) → String
  x : bigfloat
Converts bigfloats to Strings of the form "significand*2^exponent".

Examples:

  > (bigfloat-fields->string (make-bigfloat 2 -1))

  - : String

  "2*2^-1"

  > (bigfloat-fields->string (bf 1))

  - : String

  "170141183460469231731687303715884105728*2^-127"

(string->bigfloat s) → (U bigfloat False)
  s : String
Converts Strings to bigfloats. If s doesn’t parse, string->bigfloat returns #f.

Examples:

  - : (U False bigfloat)

(bfulp x) → Exact-Rational

Returns the value of the unit in last place, or the value of the lowest significant bit in the significand.

Think of an ulp as the smallest change that can possibly be made to a bigfloat without changing its exponent. To actually make these “smallest changes,” use bfnext and bfprev.

  (bf #e7.34683969263929692480460335763903548637e-40)

  - : bigfloat

  (bf #e7.34683969263929692480460335763903548637e-40)

To measure floating-point error in ulps, use exact-ulp-error, bfulp-error and bftest-error.

4.5 Cached constants

(define-bf-constant id e ...+)
Defines a new constant. Constants defined using define-bf-constant are cached at certain precisions and rounded to target precision when asked for.

Examples:

  > (define-bf-constant bf-1/7 (bf 1/7))
  > (bf-bits 64)
  > (bf-1/7)

  - : bigfloat

  (bf #e0.14285714285714285714)

  > (bf-bits 128)
  > (bf-1/7)

  - : bigfloat

  (bf #e0.1428571428571428571428571428571428571426)

In addition to binding id, for debugging purposes, define-bf-constant binds id-cache to the cache. To allow a little manual cache management, it also binds id-clear to a zero-argument procedure that clears id’s cache.

Examples:

  > bf-1/7-cache

  - : (HashTable Integer bigfloat)

  '#hash((6 . (bf #e0.14285714285714285714285)) (7 . (bf #e0.1428571428571428571428571428571428571428575)))

  > (bf-1/7-clear)
  > bf-1/7-cache

  - : (HashTable Integer bigfloat)

  '#hash()

Don’t write programs that depend on id-cache’s type or contents, or on how or when id populates the cache.

  x : bigfloat
  y : bigfloat
Computes the arithmetic-geometric mean of x and y. Typically, this isn’t directly useful, but it is integral to some asymptotically fast algorithms such as the one that computes bflog.

Some Bigfloat functions, like bf>, use bfceiling-log2 and bffloor-log2 to make fast, coarse-grained semidecisions before making relatively computationally intensive, fine-grained decisions.
(bfceiling-log2 x) → Integer
  x : bigfloat
(bffloor-log2 x) → Integer
  x : bigfloat
Equivalent to (bigfloat->integer (bfceiling (bflog2 x))) and (bigfloat->integer (bffloor (bflog2 x))), respectively, but much, much faster.

4.7 Randomness and statistics

(bfrandom) → bigfloat
(bfrandom-signed) → bigfloat
Returns uniformly distributed random bigfloats in the ranges [0,1] and [-1,1], respectively.

4.8 Measuring error

(exact-ulp-error x r) → Exact-Rational
  x : bigfloat
  r : Exact-Rational
Returns the absolute difference, in ulps with respect to x, between x and r. (For the definition of ulp, see bfulp.)

Because Bigfloat functions round correctly, the following examples return error less than 0.5 ulps.

Examples:

  > (bf (exact-ulp-error (bf/ (bf 1) (bf 7)) 1/7))

  - : bigfloat

  (bf #e0.285714285714285714285714285714285714285)

  > (bf (exact-ulp-error (bf* (bf 1/7) (bf 14/13))
                         (* (bigfloat->rational (bf 1/7))
                            (bigfloat->rational (bf 14/13)))))

  - : bigfloat

  (bf #e0.406593406593406593406593406593406593406)

However, Bigfloat functions can’t guarantee that compositions introduce no more than 0.5 ulps error.

Example:

  > (bf (exact-ulp-error (bf* (bf 1/7) (bf 14/13))
                         (* 1/7 14/13)))

  - : bigfloat

  (bf #e0.538461538461538461538461538461538461539)

(bfulp-error x y) → bigfloat
  x : bigfloat
  y : bigfloat
Returns the absolute difference, in ulps with respect to x, between x and r. (For the definition of ulp, see bfulp.) Assuming that y is exact, the result itself is correct within 0.5 ulps. To ensure the result means something, bfulp-error requires that y have at least as much precision as x.

Use this when it isn’t possible to test the accuracy of x against an exact value.

Examples:

  > (bfulp-error (bfexp (bf 2))
                 (parameterize ([bf-bits  (* 4 (bf-bits))])
                   (bfexp (bf 2))))

  - : bigfloat

  (bf #e0.1400030542059236249125258148006923429465)

  > (define x (bflog (bf 2)))
  > (bfulp-error (bfexp x)
                 (parameterize ([bf-bits  (* 2 (bf-bits))])
                   (bfexp x)))

  - : bigfloat

  (bf #e0.1268559441025884233631408847626516470496)

(bftest-error expr)
(bftest-error expr bits)
Evaluates expr at (bf-bits) precision. The first form then evaluates expr at (* 2 (bf-bits)) precision, and returns the difference in ulps as given by bfulp-error. The second form evaluates expr the second time at bits precision.

Examples:

  > (bftest-error (bfexp (bf 2)))

  - : bigfloat

  (bf #e0.1400030542059236249125258148006923429465)

  > (bftest-error (bfexp (bf 2)) (* 4 (bf-bits)))

  - : bigfloat

  (bf #e0.1400030542059236249125258148006923429465)

(bftest-error (bfexp (bf 2)) (* 4 (bf-bits))) is equivalent to the first example in bfulp-error.

4.9 Low-level functions

(new-bigfloat sig exp) → bigfloat
  sig : Integer
  exp : Integer
An alternative to the bigfloat constructor. In contrast to make-bigfloat, new-bigfloat ensures that the constructed value has exactly (bf-bits) bits precision.

Examples:

  > (bf-bits 4)
  > (make-bigfloat 31 0)

  - : bigfloat

  (bf #e31.0)

  > (new-bigfloat 31 0)

  - : bigfloat

  (bf #e32.0)

In this example, 31 has been rounded up to 32 because 31 doesn’t fit in a four-bit float. On the other hand, 32 does, as (bigfloat 8 2).

(integer->bigfloat i) → bigfloat
  i : Integer
(float->bigfloat f) → bigfloat
  f : Float
(rational->bigfloat r) → bigfloat
  r : Exact-Rational
(real->bigfloat x) → bigfloat
  x : Real
Convert real numbers to bigfloats; real->bigfloat dispatches to the other three.

These conversion functions are for completeness. It’s shorter and easier to use bf. It’s not noticeably slower, especially given the computation done in most Bigfloat functions.

(bfnext x [n]) → bigfloat
  x : bigfloat
  n : Integer = 1
(bfprev x [n]) → bigfloat
  x : bigfloat
  n : Integer = 1
Return the next largest or next smallest bigfloat with the same exponent.

Examples:

  > (bfnext (bf 1))

  - : bigfloat

  (bf #e1.000000000000000000000000000000000000006)

  > (bfulp-error (bf 1) (bfnext (bf 1)))

  - : bigfloat

  (bf #e1.0)

These functions don’t necessarily return values with (bf-bits) precision. Additionally, they can (rarely) return values with different precisions than their inputs.

(bfnormalize x) → bigfloat
  x : bigfloat
Returns the closest bigfloat with (bf-bits) precision.

Examples:

  > (bf-bits 128)
  > (define x (bf 1/7))
  > x

  - : bigfloat

  (bf #e0.1428571428571428571428571428571428571426)

  > (bf-bits 64)
  > (bfnormalize x)

  - : bigfloat

  (bf #e0.14285714285714285714)

(with-bf-bits bits exprs ...+)
Computes exprs with bits precision, then converts the result to the outer precision. Equivalent to (bfnormalize (parameterize ([bf-bits bits]) exprs ...)).

Examples:

  > (bf/ (bfsin (bf 12)) (bfcos (bf 12)))

  - : bigfloat

  (bf #e-0.635859928661580792460937942939007606149)

  > (with-bf-bits (* 2 (bf-bits))
      (bf/ (bfsin (bf 12)) (bfcos (bf 12))))

  - : bigfloat

  (bf #e-0.635859928661580792460937942939007606152)

  > (with-bf-bits (add1 (bf-bits))
      (bf/ (bfsin (bf 12)) (bfcos (bf 12))))

  - : bigfloat

  (bf #e-0.635859928661580792460937942939007606152)

(bfcanonicalize x) → bigfloat
  x : bigfloat
When hashing bigfloats, the Bigfloat library canonicalizes them first to ensure that equality implies an equal hash. Returns a new bigfloat with no more bits of precision than are necessary to encode x exactly, by removing all low-order zeros from the significand and adjusting the exponent. Two bigfloats are equal? (equivalently bf=) if and only if their canonicalized forms’ struct fields are equal.

Canonicalizing bigfloats shouldn’t change answers computed from them, but may change how they convert to strings.

Examples:

  > (bf2^ -2)

  - : bigfloat

  (bf #e0.25)

  > (equal? (bf2^ -2) (bfcanonicalize (bf2^ -2)))

  - : Boolean

  #t

  > (bfcanonicalize (bf2^ -2))

  - : bigfloat

  (bf #e0.2)

This function doesn’t necessarily return a value with (bf-bits) precision.
